Regras do sistema

Esta página explica como o sistema toma decisões para reduzir esforço mental no dia a dia. No Dashboard, você vê as tarefas mudando de estado (pendente, feita, adiada, bloqueada) conforme o contexto. Nesta fase (3.1), o foco é tornar as decisões mais claras e confiáveis — sem “mágica escondida”.

Como o sistema decide

  • Você não organiza apenas tarefas. O sistema ajuda a organizar decisões sob restrição.
  • Ele lê os atributos de cada tarefa (prioridade, tipo, categoria, dependência) e aplica regras.
  • O fluxo geral é: ler → aplicar regras → atualizar estados → mostrar na tela → salvar estado.

Objetivo: quando o dia ficar apertado, você não precisa “recalcular tudo na cabeça”.

O que significam os status

  • Pendente: tarefa planejada para o período, ainda não concluída.
  • Feita: tarefa concluída (o sistema atualiza o progresso automaticamente).
  • Adiada: tarefa não é essencial naquele momento e pode ser movida para depois.
  • Bloqueada: existe um pré-requisito antes de executar essa tarefa.

Importante: “bloqueada” não é “proibida”. É só “ainda não faz sentido agora”.

O que é “período comprometido”

  • Um período comprometido significa: menos tempo/energia disponível.
  • Quando isso acontece, o sistema tenta reduzir ruído, adiando tarefas variáveis de baixa prioridade.
  • Hoje isso aparece como um estado do período (ex.: “tarde comprometida”) que dispara regras de adaptação.

Exemplo real: “tive reunião inesperada / deslocamento / urgência” → o sistema simplifica o plano.

Ordenação padrão

  • As tarefas são organizadas primeiro por período do dia (manhã, tarde, noite).
  • Dentro do período, tarefas de maior prioridade vêm primeiro.
  • Tarefas do tipo fixa tendem a ter precedência sobre tarefas variáveis.

A ideia é começar pelo que é essencial antes do “opcional”.

Regras de adaptação (quando o dia muda)

  • Se um período estiver cheio ou comprometido, tarefas variáveis de baixa prioridade podem mudar para adiada.
  • Se uma tarefa tiver dependência (pré-requisito), ela pode ficar com status bloqueada.
  • Se surgir um imprevisto, o sistema deve permitir reorganizar o período sem perder o registro do que ficou para depois.

Nesta fase, a adaptação já acontece; o refinamento agora é tornar o “por quê” sempre visível.

Transparência: por que o sistema fez isso?

  • Se bloqueou: deve ficar claro qual é o pré-requisito (ex.: “depende de alongar”).
  • Se adiou: deve ficar claro qual foi o contexto (ex.: “período comprometido”).
  • Se reordenou: deve ficar claro qual foi o critério (prioridade e tipo).

Regra de UX: o sistema pode decidir, mas não pode “mexer escondido”.

Regra de consistência (para não virar bagunça)

  • Tarefas fixas não devem simplesmente “sumir”.
  • Se não puderem ser executadas no período original, podem mudar de estado (ex.: adiada), mas permanecem registradas.
  • O sistema deve evitar o “cemitério de tarefas” (adiar eternamente) — isso será tratado nas próximas evoluções.

Controle do usuário (evolução planejada)

  • Se o usuário discordar de uma decisão (ex.: não quer adiar), ele deve poder intervir.
  • Exemplos de controles futuros: “executar mesmo assim”, “reverter decisão”, “ignorar regra hoje”.

Produto bom = regra clara + autonomia. O sistema ajuda, mas não manda.
